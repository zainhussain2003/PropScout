### Fix 1 — RentalCompsBar: aria-label on div with no role

**Suite:** Vitest
**Test:** `RentalCompsBar > has no axe accessibility violations`
**Failure message:**

```
Error: expect(received).toHaveNoViolations(expected)
Expected the HTML found at $('div[aria-label="Estimated rent: $2,900"]') to have no violations:
"Elements must only use permitted ARIA attributes (aria-prohibited-attr)"
Fix all of the following:
  aria-label attribute cannot be used on a div with no valid role attribute.
```

**Root cause:** The diamond marker `<div>` had `aria-label` but no `role`, violating WCAG `aria-prohibited-attr`.
**Fix applied:** `apps/web/src/components/analysis/RentalCompsBar.tsx` — added `role="img"` to the diamond `div`.
**Fix type:** Component fix

### Fix 6 — Loading state: regex too broad, matched loading-state paragraph

**Suite:** Vitest
**Test:** `InvestorReport page — loading state > does NOT render section §01 while loading`
**Failure message:**

```
Error: expect(element).not.toBeInTheDocument()
expected document not to contain element, found <p ...>Calculating investment metrics…</p>
```

**Root cause:** The test used `queryByText(/investment metrics/i)` (case-insensitive regex). The LoadingState component renders `<p>Calculating investment metrics…</p>` which contains "investment metrics" as a substring, causing a false positive match. The test intent was to verify the §01 _section heading_ "Investment metrics" was absent — but the regex accidentally matched the loading paragraph instead.
**Fix applied:** `Week3-4 Front end/PR4/investor.page.test.tsx` — changed `queryByText(/investment metrics/i)` → `queryByText('Investment metrics')` (exact, case-sensitive). The loading paragraph has a lowercase 'i' and longer text, so it no longer matches. Only the §01 SectionHead topic span matches exactly.
**Fix type:** Test expectation corrected — the regex was overly broad and matched unintended text.
